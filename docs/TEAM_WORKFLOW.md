# دليل العمل المشترك لفريق 4 أشخاص
## إعداد المشروع للأول مرة

### الخطوة 1: تهيئة المشروع من أي عضو في الفريق
```bash
# أنشئ مجلد المشروع
mkdir BWP401
cd BWP401

# ابدأ مستودع Git
git init

# أضف ملف README.md
echo "# BWP401 مشروع المجموعة" > README.md
echo "## أعضاء الفريق" >> README.md
echo "- عضو 1" >> README.md
echo "- عضو 2" >> README.md
echo "- عضو 3" >> README.md
echo "- عضو 4" >> README.md

# ربط بالـ Repository البعيد
git remote add origin https://github.com/hoz2syr/BWP401.git

# أول دفع للمشروع
git add .
git commit -m "init: تهيئة المشروع الأولية"
git branch -M main
git push -u origin main
```

### الخطوة 2: توزيع الأدوار الموصى بها لـ 4 أشخاص
| الدور | المسؤوليات |
|-------|------------|
| **قائد الفريق** | إدارة المشروع، مراجعة النهائية، حل النزاعات، دمج الطلبات |
| **مطور 1** | بناء الواجهة الأمامية (Frontend) والتصميم |
| **مطور 2** | بناء المنطق الخلفي (Backend) وقواعد البيانات |
| **مختبر ومدير جودة** | كتابة الاختبارات، التأكد من جودة الكود، التوثيق |

## سياسة الفروع (Branching Strategy)
نستخدم نمط GitHub Flow البسيط والمناسب للمشاريع الجامعية:

1. **`main`** - الفرع الرئيسي، دائماً مستقر ومجهز للعرض
2. **فروع الميزات** - كل مهمة جديدة في فرع منفصل

### تسمية الفروع الصحيحة:
```
feature/اسم-الميزة
fix/اسم-الاصلاح
docs/اسم-التحديث
```

### مثال:
```bash
# إنشاء فرع جديد لمهمتك
git checkout -b feature/login-page

# بعد الانتهاء من العمل
git add .
git commit -m "feature: اضافة صفحة تسجيل الدخول"
git push origin feature/login-page
```

## سير العمل اليومي لكل عضو في الفريق

### الخطوات التي يجب اتباعها *قبل* البدء بالعمل يومياً:
```bash
# 1. التأكد من أنك على الفرع الرئيسي
git checkout main

# 2. جلب أحدث التغييرات من الفريق
git pull origin main

# 3. إنشاء فرع جديد لمهمتك اليومية
git checkout -b feature/your-task-name
```

### بعد الانتهاء من المهمة:
```bash
# 1. حفظ التغييرات
git add .
git commit -m "وصف واضح لما قمت به"

# 2. جلب آخر التحديثات من main مرة أخرى لتجنب التعارضات
git pull origin main

# 3. حل أي تعارضات إن وجدت

# 4. دفع فرعك إلى GitHub
git push origin feature/your-task-name
```

## عملية المراجعة والدمج (Pull Request)
1. بعد دفع فرعك، اذهب إلى صفحة المشروع على GitHub
2. انقر على **"Compare & pull request"**
3. أكمل تفاصيل الطلب:
   - عنوان واضح للمهمة
   - وصف تفصيلي لما تم إنجازه
   - قم بإضافة باقي أعضاء الفريق كمراجعين

### قواعد المراجعة:
✅ **يجب أن يوافق على الأقل عضو واحد آخر في الفريق قبل الدمج**  
✅ لا تدمج طلبك بنفسك بدون موافقة الآخرين  
✅ كل طلب يجب أن يمر على الاختبارات بنجاح

## اتفاقيات كتابة الـ Commit Messages
استخدم هذا النمط الموحد لسهولة تتبع التغييرات:

```
<النوع>: <الوصف>

مثال:
feature: اضافة نظام تسجيل الدخول
fix: اصلاح خطأ في عرض الاسعار
docs: تحديث دليل التثبيت
style: تعديل تصميم الازرار
test: اضافة اختبارات لصفحة الدفع
```

## إعداد المستودع على GitHub
لمنع الأخطاء، قم بإعداد هذه القواعد من قبل قائد الفريق:

1. اذهب إلى Settings → Branches → Add rule
2. حماية الفرع `main`:
   - ✅ Require a pull request before merging
   - ✅ Require approvals (1 موافقة على الأقل)
   - ✅ Include administrators
   - ✅ Restrict who can push to matching branches

## نصائح العمل الجماعي الناجحة
1. **تواصل يومياً**: قم بإعلام الفريق قبل البدء بمهمة جديدة لتجنب العمل المتكرر
2. **الفروع الصغيرة**: لا تضع كل عملك في فرع واحد كبير - قسم المهمات لقطع صغيرة
3. **التعليقات**: اكتب تعليقات واضحة في الكود ليفهمها باقي الفريق
4. **الاجتماعات**: اجتماع يومي قصير (10-15 دقيقة) لمناقشة التقدم
5. **التعارضات**: حل التعارضات بشكل هادئ مع الفريق ولا تدمج دون اتفاق

---

## خلاصة خطوات كل عضو:
1. `git pull` من main قبل البدء
2. أنشئ فرع جديد لكل مهمة
3. اكتب الكود والاختبارات
4. `git pull origin main` مرة أخيرة
5. دفع الفرع وإنشاء Pull Request
6. انتظر موافقة عضو آخر
7. قائد الفريق يقوم بالدمج إلى main
